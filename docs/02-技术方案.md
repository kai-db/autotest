# 技术方案：Claude Code + mobile-mcp + autotest

> 通过 Claude Code 连接真机，用自然语言驱动 Android App 自动化测试。

## 1. 方案概述

### 目标

开发者/测试人员用中文对 Claude Code 说"帮我测一下 DeBox 的登录流程"，Claude Code 自动：
1. 连接真机
2. 启动 DeBox App
3. 执行登录操作（输入账号、点击按钮）
4. 截图验证每一步结果
5. 输出测试结论

同时，发现的 bug 可以固化为 autotest 框架的 JUnit 用例，加入 CI 自动回归。

### 核心思路

**不在 Android 端内置 AI**，而是让 Claude Code 作为外部 AI 大脑，通过 mobile-mcp 远程操控手机。

```
你说话 → Claude Code 想 → mobile-mcp 做 → 手机动 → Claude Code 看截图 → 判断对错
```

## 2. 技术选型

### 为什么用 mobile-mcp 而不是在框架里调 AI API？

| 方案 | 优点 | 缺点 |
|---|---|---|
| **框架内调 AI API**（已废弃） | 端到端自包含 | 需要 Android 端发 HTTP、AI 看不到屏幕、无法灵活交互 |
| **Claude Code + mobile-mcp**（采用） | AI 能看截图、灵活对话、无需改 App 代码 | 依赖 MCP 连接稳定性、执行速度较慢 |

### mobile-mcp 能力清单

| 工具 | 能力 | 用途 |
|---|---|---|
| `mobile_list_available_devices` | 列出连接的设备 | 确认设备就绪 |
| `mobile_launch_app` | 启动 App | 启动 DeBox |
| `mobile_terminate_app` | 关闭 App | 清理环境 |
| `mobile_take_screenshot` | 截图 | 每步验证 |
| `mobile_list_elements_on_screen` | 获取页面元素树 | 找到按钮/输入框 |
| `mobile_click_on_screen_at_coordinates` | 坐标点击 | 点击按钮 |
| `mobile_type_keys` | 输入文本 | 填写账号密码 |
| `mobile_swipe_on_screen` | 滑动 | 翻页、下拉刷新 |
| `mobile_press_button` | 按键（返回/Home） | 导航 |
| `mobile_get_screen_size` | 获取屏幕尺寸 | 计算滑动坐标 |
| `mobile_install_app` / `mobile_uninstall_app` | 安装/卸载 | 环境准备 |

## 3. 详细实现方案

### 3.1 环境准备

```
前置条件：
1. Android 手机通过 USB 连接 Mac，adb 可用
2. mobile-mcp server 已配置并运行
3. DeBox App 已安装在手机上
4. Claude Code 已配置 mobile-mcp 工具
```

验证命令：
```bash
adb devices                    # 确认设备连接
claude                         # 启动 Claude Code
# 在 Claude Code 中：
# "列出可用的移动设备"  → 调用 mobile_list_available_devices
```

### 3.2 AI 交互式测试流程

#### 单步执行模式

用户每说一句，Claude Code 执行一步：

```
用户：启动 DeBox
Claude：[调用 mobile_launch_app] → [截图确认] → "已启动，当前在首页"

用户：点击底部的"我的"Tab
Claude：[list_elements 找到坐标] → [click] → [截图] → "已切换到我的页面"

用户：检查下登录状态
Claude：[截图] → [分析截图] → "当前未登录，显示登录按钮"
```

#### 自动执行模式

用户描述完整场景，Claude Code 自动规划并执行：

```
用户：测试 DeBox 的手机号登录流程，用 13800000001 登录

Claude：
1. [launch_app] 启动 DeBox → [截图] ✓ 在首页
2. [click] 点击"我的" → [截图] ✓ 我的页面
3. [click] 点击"登录" → [截图] ✓ 登录页
4. [list_elements] 找到手机号输入框
5. [type_keys] 输入 13800000001 → [截图] ✓ 已输入
6. [click] 点击"获取验证码" → [截图] ✓ 已发送
7. 等待用户提供验证码...
8. [type_keys] 输入验证码
9. [click] 点击"登录"
10. [截图] 验证登录成功 → 显示用户昵称 ✓

测试结果：登录流程正常，耗时约 15 秒
```

### 3.3 截图驱动的验证

关键设计：**关键节点截图**（不是每步都截），Claude Code 通过视觉理解判断：
- 页面是否正确跳转
- 是否出现错误提示/弹窗
- 文本内容是否符合预期
- 布局是否异常

截图时机：用例开始时、页面跳转后、验证点、操作失败时。连续的简单操作（如多次点击）不需要每次都截图。

```
操作 → [关键节点] 截图 → AI 分析 → 判定 → 下一步
```

### 3.4 bug 固化为 CI 用例

当 AI 测试发现 bug 时，可以将操作步骤转化为 autotest 框架的 JUnit 用例：

```
AI 发现：点击"社区" Tab 后页面白屏

生成 autotest 用例：
@Test
fun testCommunityTab_shouldLoadContent() {
    val s = scenario("社区Tab加载") {
        step("启动App") { launchAppAndDismissDialogs() }
        step("点击社区Tab") { device.clickText("社区") }
        step("等待内容加载") { device.waitForText("推荐", 5000) }
        step("验证内容可见") {
            AppAssertions.assertTextVisible(device, "推荐")
        }
    }
    s.run()
}

加入 CI → 每次发版自动回归，防止问题复现
```

### 3.5 与 autotest 框架的分工

```
┌──────────────────────┐     ┌──────────────────────┐
│  Claude Code          │     │  autotest 框架        │
│  + mobile-mcp         │     │  + Gradle + CI        │
│                       │     │                       │
│  探索性测试            │     │  回归测试              │
│  新功能验证            │  →  │  核心路径保护           │
│  bug 复现             │ 固化 │  发版前自动检查         │
│  UI 视觉检查          │     │  Flaky 自动重试        │
│  灵活、慢、智能        │     │  固定、快、稳定         │
└──────────────────────┘     └──────────────────────┘
```

## 5. 铁律：AI 监工模式 — 迭代修复闭环

> 这是本方案的核心执行流程。AI 全程自主驱动，编译→测试→修复→回归，循环直到 0 个 FAIL。

### 5.1 流程总览

```
══════════════════════════════════════════════════════════
  迭代修复阶段（循环，直到 0 个 FAIL）
══════════════════════════════════════════════════════════

  Phase 1: 编译部署
  gradlew assembleDebug → adb install → 确认 App 启动正常
      │
      ▼
  Phase 2: 全量测试
  按 P0→P1→P2 遍历 TEST_CASES.md 全部用例
  每条：重置环境 → 执行操作 → 等待 → 验证(截图+元素+状态)
      │
      ▼
  Phase 3: 更新文档
  将本轮全部结果写入 TEST_RESULTS.md（PASS / FAIL / bug 记录）
      │
      ▼
  Phase 4: 全部修复
  汇总本轮所有 FAIL，逐个分析根因，逐个修复代码
  修复记录写入 TEST_RESULTS.md
      │
      ▼
  Phase 5: 回归测试
  重新编译部署（回到 Phase 1）
  重跑全部用例（不只是 FAIL 的，防止修复引入新 bug）
      │
      ├── 仍有 FAIL → 回到 Phase 3 更新文档，Phase 4 继续修复
      │
      └── 全部 PASS → 进入最终验收阶段

══════════════════════════════════════════════════════════
  最终验收阶段（全量跑一遍，确认稳定）
══════════════════════════════════════════════════════════

  Phase 6: 全量验收跑
  不做任何代码修改，从 Phase 1 开始完整执行一遍
  全部用例逐条跑，每条都要重置+验证
      │
      ├── 全部 PASS → ✅ 测试通过，更新最终文档，流程结束
      │
      └── 有 FAIL → 回到迭代修复阶段继续
```

### 5.2 各 Phase 详细操作

#### Phase 1: 编译部署

```bash
# 1. 编译 Debug APK
cd /path/to/project
./gradlew assembleDebug

# 2. 安装到设备
mobile_install_app  device: "设备ID"  path: "app/build/outputs/apk/debug/app-debug.apk"

# 3. 启动 App 确认安装成功
mobile_launch_app  device: "设备ID"  packageName: "包名"

# 4. 截图确认 App 正常启动
mobile_take_screenshot
```

#### Phase 2: 全量测试

按优先级执行 TEST_CASES.md 中的用例：

```
对于每条用例：
  1. 重置环境
     - mobile_terminate_app（杀掉 App）
     - mobile_launch_app（重新启动）
     - 处理弹窗（权限/升级/引导）

  2. 执行操作步骤
     - 按用例定义的步骤逐步操作
     - 关键节点截图确认

  3. 验证三件套
     - 截图验证：页面视觉是否正确
     - 元素验证：关键元素是否存在（list_elements）
     - 状态验证：App 是否在前台、是否崩溃

  4. 记录结果
     - PASS：验证全部通过
     - FAIL：记录失败步骤 + 截图 + 错误描述
```

#### Phase 3: 更新文档

每轮测试完成后，立即更新 TEST_RESULTS.md：

```markdown
# 测试结果 — 第 N 轮

| 用例编号 | 用例名称 | 优先级 | 结果 | 失败原因 | 截图 |
|---|---|---|---|---|---|
| TC-001 | 冷启动 | P0 | PASS | - | - |
| TC-002 | Tab导航 | P0 | FAIL | 社区Tab白屏 | screenshot_002.png |
| ... | ... | ... | ... | ... | ... |

**本轮统计**：PASS: X / FAIL: Y / 总计: Z
```

#### Phase 4: 全部修复

```
对于每个 FAIL 用例：
  1. 分析根因
     - 读取截图和错误描述
     - 定位相关代码（grep/read）
     - 判断是代码bug、配置问题、还是环境问题

  2. 修复代码
     - 修改源码
     - 不做超出修复范围的改动

  3. 记录修复
     - 在 TEST_RESULTS.md 中记录修复内容
     - 关联的 FAIL 用例编号
```

#### Phase 5: 回归测试

```
关键原则：重跑全部用例，不只是 FAIL 的

原因：修复 A 的代码可能破坏 B 的功能

执行：回到 Phase 1 重新编译 → Phase 2 全量测试
```

#### Phase 6: 最终验收

```
前置条件：迭代修复阶段已达到 0 个 FAIL

执行：
  1. 不做任何代码修改
  2. 从 Phase 1 开始完整执行
  3. 全部用例逐条跑
  4. 全部 PASS → 流程结束
  5. 有 FAIL → 说明修复不稳定，回到迭代修复阶段
```

### 5.3 铁律（不可违反）

1. **AI 驱动自动化测试** — AI 全程自主执行，不需要人工干预（验证码等特殊情况除外）
2. **拿到测试用例后 AI 实现相关逻辑进行测试** — TEST_CASES.md 是唯一的用例来源，AI 按用例执行
3. **AI 自动开启监工模式** — 迭代修复循环自动运转，直到 0 个 FAIL
4. **最小人工介入** — 尽量减少人为操控，除验证码等必须人工提供的环节外全部 AI 自主完成
5. **修复必须合规** — 修复代码必须合理、符合项目编码规范、不引入新 Bug；修复前先理解根因，不用 hack 绕过
6. **每条用例必须重置环境** — 用例之间不能有状态依赖
7. **回归必须全量跑** — 修复后不能只跑失败的用例，全量回归就是为了验证修复不引入新问题
8. **验证必须有截图证据** — 不能凭猜测判定 PASS/FAIL

### 5.4 文件约定

| 文件 | 用途 | 谁写 | 谁读 |
|---|---|---|---|
| `TEST_CASES.md` | 测试用例定义（步骤+验证标准） | 人/AI | AI |
| `TEST_RESULTS.md` | 每轮测试结果（PASS/FAIL+截图） | AI | 人/AI |
| `screenshots/` | 测试截图目录 | AI | 人/AI |
