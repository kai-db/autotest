# 可行性评估

> Claude Code + mobile-mcp + autotest 自动化测试方案 | v1.5.1

## 1. 结论

**方案整体可行**，但有明确的能力边界。

| 维度 | 评估 | 说明 |
|---|---|---|
| 技术可行性 | ✅ 高 | mobile-mcp 已有完整的设备操控能力，Claude Code 能理解截图 |
| 稳定性 | ⚠️ 中 | 依赖 adb 连接和 MCP 通信，存在超时/断连风险 |
| 效率 | ⚠️ 中 | 每步截图分析约 3-5 秒，完整流程较慢，适合探索而非大批量 |
| 准确性 | ✅ 高 | Claude 视觉理解能力强，判断 UI 状态准确率高 |
| 成本 | ⚠️ 中 | 每次测试消耗 token，适合关键流程验证，不适合大规模轰炸 |

## 3. 风险分析

### 风险 1：adb 连接不稳定

**概率**：中
**影响**：测试中断
**现象**：USB 断连、WiFi adb 超时、设备休眠
**缓解**：
- 使用 USB 有线连接（优于 WiFi adb）
- 关闭设备自动休眠
- 测试前先 `mobile_list_available_devices` 确认

### 风险 2：元素定位不准确

**概率**：中
**影响**：点击错误位置
**现象**：`mobile_list_elements_on_screen` 返回的坐标与实际不符（动画中、页面未加载完）
**缓解**：
- 每步操作后等待 1-2 秒再截图
- 截图确认状态后再执行下一步
- 用截图视觉验证替代纯坐标信任

### 风险 3：验证码/登录态问题

**概率**：高
**影响**：无法完成登录测试
**现象**：需要真实短信验证码，AI 无法获取
**缓解**：
- 使用测试环境固定验证码
- 或人工提供验证码给 Claude Code
- 非登录测试可用已登录的设备跳过

### 风险 4：测试执行速度慢

**概率**：确定
**影响**：不适合大批量测试
**现象**：每步操作 + 截图 + AI 分析约 3-5 秒，12 步流程约 1 分钟
**缓解**：
- AI 交互式测试定位于探索性/验证性测试，不追求速度
- 大批量回归用 autotest 框架（Gradle 执行，秒级）
- 两条路线互补

### 风险 5：token 成本

**概率**：确定
**影响**：频繁截图分析消耗较多 token
**缓解**：
- 截图仅在关键步骤获取，不需要每秒截图
- 简单操作（如点击已知坐标）不需要截图验证
- 发现的 bug 固化为 autotest 用例后，后续回归零 token 成本

## 4. 能力边界

### 能做的

- ✅ 功能流程验证（启动、登录、导航、基本操作）
- ✅ UI 视觉检查（布局异常、白屏、崩溃）
- ✅ 探索性测试（随机浏览、边界操作）
- ✅ 弹窗处理（权限、升级、引导）
- ✅ 截图对比（前后状态变化）
- ✅ 生成测试报告（文字描述 + 截图证据）

### 做不了或做不好的

- ❌ 性能测试（帧率、内存、启动耗时精确测量）
- ❌ 大规模并发测试（一次只能操控一台设备）
- ❌ 需要真实短信/推送的测试（除非人工辅助）
- ❌ 后台服务/数据库层面的验证
- ⚠️ 动画过程中的精确操作（需等待动画结束）
- ⚠️ WebView 内部元素操作（element list 可能不完整）

## 5. 与纯 autotest 方案对比

| 维度 | 纯 autotest（CI） | Claude Code + mobile-mcp | 组合使用 |
|---|---|---|---|
| 编写成本 | 高（写 Kotlin 代码） | 低（说中文就行） | 最优 |
| 执行速度 | 快（秒级） | 慢（分钟级） | 互补 |
| 灵活性 | 低（固定用例） | 高（随意测） | 互补 |
| 稳定性 | 高 | 中 | 互补 |
| 视觉判断 | 无（只能断言属性） | 强（能看懂截图） | 互补 |
| CI 集成 | 原生支持 | 不适合 | autotest 承担 |
| 发现新 bug | 弱（只跑写过的用例） | 强（AI 探索） | AI 发现 → 固化 |
