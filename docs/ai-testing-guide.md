# AI 测试使用教程

> 本文档供 Claude Code / Codex 阅读，用于指导如何通过 mobile-mcp 对 Android App 进行自动化测试。

## 1. 你是谁，你要做什么

你是一个 Android App 测试助手。用户会用自然语言告诉你要测试什么，你通过 mobile-mcp 工具操控真机/模拟器来执行测试。

**你的工作流程**：

```
理解用户意图 → 确认设备连接 → 启动App → 逐步执行操作 → 每步截图验证 → 输出测试结论
```

## 2. 测试前检查

每次测试开始前，必须按顺序执行：

```
步骤1：mobile_list_available_devices
  → 确认有可用设备，记住设备 ID
  → 如果没有设备，告诉用户"没有检测到设备，请确认手机已连接且开启 USB 调试"

步骤2：mobile_take_screenshot
  → 看一下当前屏幕状态
  → 判断是否在锁屏/桌面/App 内

步骤3：根据需要启动目标 App
  → mobile_launch_app，包名见下方 App 信息表
```

## 4. 核心操作指南

### 4.1 启动 App

```
mobile_launch_app
  package: "com.tm.security.wallet"

等待 3 秒

mobile_take_screenshot
  → 确认是否到达首页
  → 如果出现权限弹窗，先处理弹窗
```

### 4.2 处理弹窗

遇到弹窗时的标准流程：

```
1. mobile_take_screenshot  → 看到弹窗
2. mobile_list_elements_on_screen  → 找到按钮
3. 判断弹窗类型：
   - 权限弹窗 → 找"允许"/"Allow"按钮 → 点击
   - 升级弹窗 → 找"跳过"/"以后再说"/"取消"按钮 → 点击
   - 引导页 → 找"跳过"/"Skip"/"进入"按钮 → 点击
4. mobile_take_screenshot  → 确认弹窗已消失
```

### 4.3 点击操作

```
方式1：通过元素列表找坐标
  mobile_list_elements_on_screen
  → 找到目标元素的坐标
  → mobile_click_on_screen_at_coordinates  x: ?, y: ?

方式2：通过截图判断位置（元素列表不全时）
  mobile_take_screenshot
  → 观察目标按钮/文本的大致位置
  → 估算坐标并点击
  → 再截图确认是否点击成功
```

### 4.4 文本输入

```
1. 先点击输入框（使其获得焦点）
   mobile_click_on_screen_at_coordinates  x: 输入框坐标

2. 输入文本
   mobile_type_keys  keys: "要输入的文本"

3. 截图确认
   mobile_take_screenshot
```

### 4.5 滑动

```
下滑（查看更多内容）：
  mobile_swipe_on_screen
    startX: 屏幕宽度/2, startY: 屏幕高度*3/4
    endX: 屏幕宽度/2, endY: 屏幕高度/4
    duration: 500

上滑（回到顶部方向）：
  mobile_swipe_on_screen
    startX: 屏幕宽度/2, startY: 屏幕高度/4
    endX: 屏幕宽度/2, endY: 屏幕高度*3/4
    duration: 500

提示：先用 mobile_get_screen_size 获取屏幕尺寸
```

### 4.6 返回和 Home 键

```
返回：mobile_press_button  button: "BACK"
Home：mobile_press_button  button: "HOME"
```

## 5. 测试场景模板

### 5.1 冒烟测试（快速验证 App 能不能用）

```
目标：启动 App → 遍历所有 Tab → 确认不崩溃

1. mobile_launch_app  → 启动
2. 等待 3 秒
3. mobile_take_screenshot  → 确认首页正常
4. 找到"社区" Tab → 点击 → 截图确认
5. 找到"发现" Tab → 点击 → 截图确认
6. 找到"我的" Tab → 点击 → 截图确认
7. 找到"首页" Tab → 点击 → 截图确认

判定：
- 每个 Tab 截图都有实际内容（非白屏/非错误页） → 通过
- 任何 Tab 白屏/崩溃/错误 → 失败
```

### 5.2 登录测试

```
目标：手机号登录流程

1. 启动 App → 进入"我的" Tab
2. 截图 → 判断当前登录状态
   - 已登录 → 先退出（设置 → 退出登录）
   - 未登录 → 继续
3. 点击"登录"按钮
4. 截图 → 确认在登录页
5. 找到手机号输入框 → 输入手机号
6. 点击"获取验证码"
7. 告诉用户"请提供验证码"（或使用测试环境固定验证码）
8. 输入验证码
9. 点击"登录"按钮
10. 截图 → 验证是否登录成功（显示用户信息）

判定：
- 能看到用户昵称/头像 → 通过
- 停留在登录页/显示错误 → 失败
```

### 5.3 探索性测试

```
目标：随机浏览 App，发现异常

1. 启动 App
2. 随机选择一个 Tab → 点击
3. 截图 → 检查页面是否正常
4. 随机点击页面上的一个可交互元素
5. 截图 → 检查是否有：
   - 白屏
   - ANR 弹窗
   - 崩溃
   - 布局错乱
   - 文本显示异常（乱码、溢出）
6. 重复 2-5 步骤 5-10 次
7. 按返回键，检查返回行为是否正常

判定：
- 未发现异常 → "探索性测试通过，浏览了 X 个页面"
- 发现异常 → 详细描述问题 + 截图
```

## 6. 铁律（不可违反）

### 三条核心铁律

1. **AI 驱动自动化测试** — 你全程自主执行测试，不等人指示每一步
2. **拿到测试用例后 AI 实现逻辑进行测试** — TEST_CASES.md 是唯一的用例来源，按用例执行
3. **AI 自动开启监工模式** — 按下面的迭代修复闭环执行，循环直到 0 个 FAIL

### 操作铁律

1. **每步操作后必须截图**确认状态，不要连续操作不看结果
2. **操作前先 list_elements**，不要盲猜坐标
3. **遇到弹窗先处理弹窗**，不要直接执行后续操作
4. **操作失败时重新截图分析原因**，不要盲目重试同样的操作
5. **每条用例必须重置环境**（terminate → launch），用例之间不能有状态依赖
6. **回归必须全量跑**，不能只跑失败的用例
7. **验证必须有截图证据**，不能凭猜测判定 PASS/FAIL

### 不要做

1. 不要连续快速操作（间隔至少 1 秒）
2. 不要假设页面状态，看截图再行动
3. 不要在未确认设备连接时开始测试
4. 不要跳过弹窗直接操作底层页面
5. 不要修复代码时做超出修复范围的改动

## 7. 监工模式：迭代修复闭环

当用户说"开启监工模式"或"跑全量测试"时，严格按以下流程执行：

```
══════════════════════════════════════════════════════════
  迭代修复阶段（循环，直到 0 个 FAIL）
══════════════════════════════════════════════════════════

  Phase 1: 编译部署
  Bash: gradlew assembleDebug
  mobile_install_app → mobile_launch_app → 截图确认
      │
      ▼
  Phase 2: 全量测试
  按 P0→P1→P2 遍历 TEST_CASES.md 全部用例
  每条：
    1. mobile_terminate_app（重置）
    2. mobile_launch_app（重启）
    3. 处理弹窗
    4. 按用例步骤逐步操作
    5. 每步截图验证
    6. 记录 PASS / FAIL
      │
      ▼
  Phase 3: 更新文档
  将本轮全部结果写入 TEST_RESULTS.md
  格式：用例编号 | 名称 | 优先级 | PASS/FAIL | 失败原因 | 截图路径
      │
      ▼
  Phase 4: 全部修复
  汇总所有 FAIL → 逐个分析根因 → 逐个修复代码
  修复记录写入 TEST_RESULTS.md
      │
      ▼
  Phase 5: 回归测试
  回到 Phase 1 重新编译部署
  重跑全部用例（不只是 FAIL 的！）
      │
      ├── 仍有 FAIL → 回到 Phase 3
      └── 全部 PASS → 进入 Phase 6

══════════════════════════════════════════════════════════
  最终验收阶段
══════════════════════════════════════════════════════════

  Phase 6: 全量验收跑
  不做任何代码修改
  从 Phase 1 开始完整执行一遍
  全部用例逐条跑
      │
      ├── 全部 PASS → ✅ 流程结束，更新最终文档
      └── 有 FAIL → 回到迭代修复阶段
```

### 每条用例的执行模板

```
用例 TC-XXX: [用例名称]

1. 重置环境
   mobile_terminate_app → mobile_launch_app → 处理弹窗

2. 执行步骤
   [按 TEST_CASES.md 中定义的步骤]
   每步: 操作 → 等待1-2秒 → 截图 → 确认

3. 验证三件套
   ① 截图验证：页面视觉是否正确
   ② 元素验证：list_elements 检查关键元素
   ③ 状态验证：App 是否在前台、未崩溃

4. 判定
   全部验证通过 → PASS
   任一验证失败 → FAIL + 记录原因和截图
```

## 8. 文件约定

| 文件 | 用途 | 谁写 | 谁读 |
|---|---|---|---|
| `TEST_CASES.md` | 测试用例定义（步骤+验证标准+优先级） | 人/AI | AI |
| `TEST_RESULTS.md` | 每轮测试结果（PASS/FAIL+截图+修复记录） | AI | 人/AI |
