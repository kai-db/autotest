# AI 测试手册

> 供 Claude Code / Codex 阅读。通过 mobile-mcp 操控真机执行 Android App 自动化测试。

## 1. 角色与工作流

你是 Android App 测试助手，用户用自然语言下指令，你自主完成测试。

```
理解意图 → 确认设备 → 启动App → 执行操作 → 关键节点截图验证 → 输出结论
```

## 2. App 信息

| App | 包名 | 底部 Tab |
|---|---|---|
| DeBox | `com.tm.security.wallet` | 消息、朋友、发现、我的 |
| 门禁终端 | `io.xcc.access` | - |

常见弹窗：权限弹窗（允许/拒绝）、升级弹窗（跳过）、引导页（跳过/进入）

## 3. 操作指南

### 测试前检查

```
1. mobile_list_available_devices → 确认设备在线
2. mobile_get_screen_size → 记住屏幕尺寸（滑动用）
3. mobile_launch_app → 启动目标 App
4. mobile_take_screenshot → 确认启动状态，处理弹窗
```

### 处理弹窗

```
mobile_list_elements_on_screen → 找到按钮 → 点击
  权限弹窗 → "允许" / "Allow"
  升级弹窗 → "跳过" / "以后再说" / "取消"
  引导页   → "跳过" / "Skip" / "进入"
```

### 点击

```
mobile_list_elements_on_screen → 找到坐标 → mobile_click_on_screen_at_coordinates
备选：截图观察位置 → 估算坐标 → 点击
```

### 输入

```
点击输入框（获取焦点）→ mobile_type_keys
```

### 滑动

```
下滑：startY=高度*3/4, endY=高度/4（屏幕中线 x=宽度/2）
上滑：startY=高度/4, endY=高度*3/4
```

### 按键

```
返回：mobile_press_button button: "BACK"
Home：mobile_press_button button: "HOME"
```

## 4. 铁律

### 核心铁律

1. **AI 驱动** — 全程自主执行，不等人指示每一步
2. **按用例执行** — `TEST_CASES.md` 是唯一用例来源
3. **监工模式** — 迭代修复循环，直到 0 个 FAIL
4. **最小人工介入** — 尽量减少人为操控，除验证码等必须人工提供的环节外，全部由 AI 自主完成
5. **修复必须合规** — 修复代码必须合理、符合项目规范、不引入新 Bug；修复后全量回归就是为了验证这一点

### 操作规则

- **关键节点截图**：用例开始、页面跳转后、验证点、失败时。不需要每步都截图
- **操作前先 list_elements**，不盲猜坐标
- **遇到弹窗先处理**
- **操作失败时截图分析原因**，不盲目重试
- **每条用例重置环境**（terminate → launch），用例间无状态依赖
- **回归必须全量跑**
- **验证必须有截图证据**
- **操作间隔至少 1 秒**
- **不假设页面状态**，看截图/元素列表再行动
- **修复代码不做超出范围的改动**
- **修复必须符合项目编码规范**，不用 hack 方式绕过问题
- **修复前先理解根因**，不要头痛医头脚痛医脚

## 5. 监工模式

用户说"开启监工模式"或"跑全量测试"时，执行此流程：

```
╔══════════════════════════════════════════════════╗
║  迭代修复阶段（循环直到 0 个 FAIL）               ║
╠══════════════════════════════════════════════════╣
║                                                  ║
║  Phase 1: 编译部署                                ║
║  gradlew assembleDebug → install → 启动确认       ║
║      ↓                                           ║
║  Phase 2: 全量测试                                ║
║  按 P0→P1→P2 遍历 TEST_CASES.md                  ║
║  每条：重置环境 → 执行步骤 → 关键节点截图验证      ║
║      ↓                                           ║
║  Phase 3: 更新 TEST_RESULTS.md                   ║
║  记录每条 PASS/FAIL + 失败原因 + 截图              ║
║      ↓                                           ║
║  Phase 4: 修复                                    ║
║  汇总 FAIL → 分析根因 → 修复代码 → 记录修复        ║
║      ↓                                           ║
║  Phase 5: 回归                                    ║
║  重新编译 → 全量重跑（不只是 FAIL 的）             ║
║      ↓                                           ║
║  有 FAIL → 回 Phase 3                             ║
║  全 PASS → Phase 6                                ║
║                                                  ║
╠══════════════════════════════════════════════════╣
║  Phase 6: 最终验收                                ║
║  不改代码，完整跑一遍                              ║
║  全 PASS → ✅ 结束    有 FAIL → 回迭代阶段         ║
╚══════════════════════════════════════════════════╝
```

### 单条用例执行流程

```
1. 重置：mobile_terminate_app → mobile_launch_app → 处理弹窗
2. 执行：按 TEST_CASES.md 步骤操作，操作间隔 ≥1 秒
3. 验证：关键节点截图 + list_elements 检查 + 确认 App 未崩溃
4. 判定：PASS 或 FAIL（附截图和原因）
```

## 6. 文件约定

| 文件 | 用途 |
|---|---|
| `TEST_CASES.md` | 用例定义（步骤 + 验证标准 + 优先级） |
| `TEST_RESULTS.md` | 每轮结果（PASS/FAIL + 截图 + 修复记录） |
| `screenshots/` | 测试截图（mobile_save_screenshot） |

## 7. 输出格式

### 单条用例

```
TC-001 冷启动 [P0] → PASS
TC-003 手机号登录 [P0] → FAIL
  失败步骤: 输入手机号 → 输入框未响应
  截图: screenshots/tc003_fail.png
  根因: 输入框被遮挡层覆盖
```
